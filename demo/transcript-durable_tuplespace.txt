# Two `fmpl-cli` REPL processes, sequenced through the same on-disk
fjall-backed TupleSpace. Alice opens the space, `out`s durable
tuples, exits. Bob — a fresh process spawned after Alice exits —
opens the same path, `rd`s the same tuples back. Byte-identical
recovery across the process boundary.

This closes the seam that the `three_hashes.yaml` demo named: the
coordination primitive itself is now persistent. Same fjall
single-writer constraint as the SourceStore demo — Alice and Bob
sequence, not overlap.

Per the ITER-0005c scope doc:
`docs/superpowers/specs/2026-05-15-iter-0005c-tuplespace-persistence.md`

Reading guide
────────────────────────────────────────────────────────────────────────

# The shape is the same as `three_hashes.yaml`:
# - Alice opens a tuplespace, writes a few tuples, exits.
# - Bob opens the same path, reads them back.
# - The harness asserts what Bob recovered equals what Alice
# stored.

# The new piece is the `durable: true` flag on each `out`. Tuples
# without it stay in memory and don't survive Alice's exit. The
# demo proves both: durable tuples come back; non-durable ones
# don't.

# source: Alice's three out() calls
```fmpl
world.out(%{type: :room, data: %{name: "Rusty Flagon"}, durable: true})
world.out(%{type: :menu, data: %{ale: 3, mead: 5}, durable: true})
world.out(%{type: :note, data: "scratch — should NOT survive"})
```
# Two durable tuples (`:room`, `:menu`) and one ephemeral note. The `durable: true` flag opts each tuple into write-through to the backing fjall keyspace; without it, the tuple lives only in memory.
[alice] fmpl> world.out(%{type: :room, data: %{name: "Rusty Flagon"}, durable: true})
=> null
[alice] fmpl> world.out(%{type: :menu, data: %{ale: 3, mead: 5}, durable: true})
=> null
[alice] fmpl> world.out(%{type: :note, data: "scratch — should NOT survive"})
=> null

────────────────────────────────────────────────────────────────────────
Bob recovers the durable tuples
────────────────────────────────────────────────────────────────────────

# `tuplespace.open` replays every `PayloadKind::Tuple` record from the keyspace into the in-memory BTreeMap, in FIFO order (big-endian seq encoding + fjall byte-ordered iter). Bob's `rd` goes through the same in-memory query path as always — the durable layer is write-through, not read-through.
[bob] fmpl> world.rd(:room).data.name
=> "Rusty Flagon"
@captured bob_room_name = "Rusty Flagon"
[bob] fmpl> world.rd(:menu).data.ale
=> 3
@captured bob_ale_price = 3

────────────────────────────────────────────────────────────────────────
Ephemeral tuple did NOT survive
────────────────────────────────────────────────────────────────────────

# The `:note` tuple was out'd without `durable: true`, so it lived only in Alice's in-memory BTreeMap. Bob's process has no knowledge of it — `rdp` errors with "no matching tuple found".
[bob] fmpl> world.rdp(:note)
Error: Runtime error: no matching tuple found
