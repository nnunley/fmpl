  FMPL — three-hash audit + cross-process source handoff
════════════════════════════════════════════════════════════════════════

# Two `fmpl-cli` REPL processes, sequenced through the same on-disk
content-addressed SourceStore. Alice defines a value, stores it under
THREE hashes (source text, runtime Value, compiled bytecode), closes
her process. Bob — a fresh process spawned after Alice exits — fetches
each hash and recovers the bytes. The harness asserts byte-identical
recovery across the process boundary.

Sequencing is required because fjall (the v3 LSM backend) takes a
single-writer file lock on the keyspace directory: two processes
cannot hold it open at the same time. Alice writes, closes (releases
the lock), Bob opens. This honest constraint is itself a finding —
the design invariant "transparent persistence" doesn't yet imply
"concurrent multi-process access," and the demo shows where that
seam will land.

Reading guide
────────────────────────────────────────────────────────────────────────

# Three artifacts derive from one FMPL binding:
# 1. SOURCE  — the exact bytes the user typed (`\x x * x`).
# 2. VALUE   — the serialized runtime Value enum (JSON).
# 3. BYTECODE — the Lambda's CompiledCode (instructions + nested).

# Each goes into the same content-addressed SourceStore under its
# blake3 hash. The store doesn't know the kind — that lives only in
# the metadata `.store-*` prints.

# Two REPL processes (Alice, Bob) sequence through one store.
# Bob's recovery is byte-identical to Alice's input — that's the
# audit at the end.

# All three round-trips succeeded across process boundaries. This is the FMPL persistence substrate working end-to-end through the shipped REPL. The fjall single-writer constraint is the real limitation today — the next iteration wires this same `Store` trait through the in-memory tuplespace, AND addresses the multi-writer story (broker process or read-replica path).
